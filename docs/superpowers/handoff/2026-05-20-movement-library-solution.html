UNBOUND Movement Library Solution

This is the proposed solution for turning the current movement audit into a durable app architecture. The key decision is to keep ExerciseCatalog as the canonical strength/gym catalog, then add a resolver layer that maps skills, drills, cardio, carry/sled work, routines, and mobility into the unified workout spine.

Recommendation

1. Add MovementCatalog as the front door.

It should contain canonical movement IDs, display names, block kind, default logger mode, stat vector, and aliases. It does not replace ExerciseCatalog; it wraps it and fills the gaps.

2. Keep skill nodes separate from exercises.

A skill node is an unlock target with standards and prerequisites. Its drills can map to movements, but the skill itself gets XP/rank logic and optional skill-node stat contribution.

3. Treat routines as containers.

Routine templates should become saved session drafts or routine blocks. Their typed steps resolve to movement prescriptions where possible.

4. Add metric-specific reward adapters.

Strength/bodyweight use sets. Skill uses attempts/holds/quality. Cardio uses duration/distance/intensity. Carry uses load/distance/duration/posture. Mobility uses duration/side/quality.

Implementation Shape

  • Phase A: introduce MovementDefinition, MovementAlias, MovementRole, MovementLoggerMode, and MovementResolver.
  • Phase B: seed resolver from existing ExerciseCatalog, CardioType, selected carry/sled entries, and a small mobility catalog.
  • Phase C: map skill drills and routine steps through aliases when building TrainingSessionDraft.
  • Phase D: move stat reward lookup from raw exercise string to resolved movement ID, preserving current exercise-name fallback during migration.
  • Phase E: convert remaining alias_needs_target rows into explicit aliases or canonical bodyweight/mobility entries.
