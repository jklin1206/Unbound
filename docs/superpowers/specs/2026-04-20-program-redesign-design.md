# UNBOUND Program Redesign — Design Spec

**Date:** 2026-04-20
**Status:** Approved design, ready for implementation plan
**Scope:** Replace the current program generation and presentation with a deterministic, tailored, always-on 2-week block model driven by scan weak-points, user-configurable training feedback, and the existing Hawks progression engine.

---

## 1. Mental Model

A program is a **continuously flowing routine** surfaced in **2-week check-in cycles**. The 2-week block is a **UX unit, not a programming reset unit** — underneath, progression, exercise selection, and weak-point bias all flow continuously. The block boundary is a scheduled beat where the user sees their progress (album, re-scan nudge, tier-unlock recap) and the app can optionally refresh the plan.

The plan itself is produced by a pure function:

```
(archetype, trainingStyle, equipment, targetFrequency, trainingDays,
 experience, latestScanWeakPoints, cutModeActive, progressionStates, previousBlock?)
  → TrainingProgram
```

No AI call per generation — the function is math plus a hand-authored exercise catalog. When a block completes, the next block is generated by the same function with fresher inputs: progression state has advanced, scan data may be newer, cut state may have toggled. The function is deterministic and *conservative* about changing things — most blocks carry forward the previous block's exercise selection and accessory bias unchanged.

**Cut mode** is a single boolean flag that toggles macro targets and switches Hawks from "advance" to "preserve." It is opt-in. Default state is always-on maintenance/build; no phase framing, no forced cut.

**The user never sees "phase" or "cycle" language.** They see: today's workout, a 2-week calendar, macros, and a progress album. The anime-arc framing comes from tier unlocks and re-scan check-ins, not from forced periodization. Block number surfaces only in the progress album as an "Arc counter."

### Research-grounded rationale

Standard hypertrophy mesocycles are 4–6 weeks in evidence-based coaching (Renaissance Periodization, Juggernaut). Exercise rotation is recommended every 4–6 weeks (6–12 for beginners). Full-body-comp changes aren't reliably detectable in 2 weeks, though visible changes (water, glycogen, early fat loss, newbie gains) absolutely are.

This informs three deliberate choices:
- **2-week cadence** is preserved as a UX/progress-check rhythm (users want and deserve the dopamine hit of visible change every 2 weeks), but it does **not** force programming shifts.
- **Accessory bias refresh** is gated on *meaningful* scan priority changes, not "every scan." If the top-2 weak-point muscles haven't changed rank, the bias carries forward.
- **Exercise refresh** is scheduled: after ~3 blocks (≈6 weeks) on the same exercise with no tier unlock and no plateau, the generator rotates it to a same-pattern alternative. This matches the 4–6 week rotation standard while keeping things stable long enough for adaptation.

## 2. Inputs → Outputs

### Inputs (read from `UserProfile` + latest `BodyAnalysis` + `ProgressionState` store)

- `archetype`
- `trainingStyleOverride?` (nil → use archetype default)
- `equipment: [Equipment]` (~7 chips, multi-select)
- `targetFrequency: 3 / 4 / 5 / 6`
- `trainingDays: Set<Weekday>` (count == targetFrequency)
- `experience: beginner / intermediate / advanced`
- `latestScan.focusAreas` + `muscleAssessments` (priority-ranked weak points)
- `cutModeActive: Bool`
- `progressionStates: [ProgressionState]` (per-exercise weight/rep carry-forward)
- `trainingFeedbackMode: silent / quick / detailed`
- `previousBlock: ProgramBlock?` (for continuity)

### Output

- A `TrainingProgram` with 14 `ProgramDay` entries, nutrition targets, recovery activities, and a populated `ProgramRationale`.
- A new `ProgramBlock` record stamped with this block's `accessoryBias`, `cutModeActive`, and `scanId`.

## 3. Data Model

### Stays as-is

- `TrainingProgram` (top-level container)
- `ProgramDay`, `Workout`, `Exercise`
- `ProgressionState`, `ProgressionFamilyState`
- `ExerciseCatalog` with families and tiers
- `BodyAnalysis`, `MuscleGroupAssessment`, `FocusArea`, `ProportionData`
- `ProgramRationale` + `WhyThisProgramView`

### Changes

- `TrainingProgram.durationDays` — locked to 14. Remove as generator input.
- `TrainingProgram.difficultyLevel` — repurpose from generator input to a derived read from `experience`.
- `ProgramDay.label` — narrative strings ("Push + Shoulder Bias") generated from day template + active weak-point bias.
- `ProgressionState.targetRPE` — driven by `UserProfile.trainingFeedbackMode`, not `blockType`. Silent mode sets target to 0 so `hitTargetRPE` always passes.
- `ProgressionState.blockType` — locked to `.accumulation` by default. Never auto-promoted to `.intensification` or `.realization`. `.deload` fires only as a reaction to plateau, per exercise, for one 2-week block.

### Added

- `ProgramBlock` — lightweight record per 2-week block:
  ```
  id, userId, programId, blockNumber, startedAt, endedAt?, scanId,
  accessoryBias: [MuscleGroup: Int], cutModeActive: Bool,
  biasRefreshedFromPrevious: Bool,  // true = carried forward, false = refreshed
  exerciseRotationsThisBlock: [String]  // exerciseKeys that rotated in this block
  ```
- `TrainingFeedbackMode: silent / quick / detailed` — on `UserProfile`. Auto-defaulted from `TrainingExperience` (beginner → silent, intermediate → quick, advanced → detailed). Editable in Settings.
- `ProgressPhoto` — daily optional photos for the album. Separate from `ScanSession`:
  ```
  id, userId, storageUrl, capturedAt, note?, angle?
  ```
- `CutMode` struct on `UserProfile`:
  ```
  { enabled: Bool, startedAt: Date?, targetDurationWeeks: Int? }
  ```
- `TrainingDays: Set<Weekday>` on `UserProfile`. Size must equal `TargetFrequency`.
- `TrainingStyleOverride: TrainingStyle?` on `UserProfile`. Nil means use archetype default.

### Removed

- Onboarding `Frequency` (current) step and enum (if unreferenced elsewhere).

## 4. Generation Pipeline

All deterministic. No LLM.

1. **Resolve style.** `style = trainingStyleOverride ?? archetype.defaultStyle`. Filter `ExerciseCatalog` to eligible pool crossed with equipment chips.

2. **Pick the split.** Lookup table keyed on `(archetype, targetFrequency)`:
   - 3 → Full-body ×3 (Calisthenic) or Upper/Lower/Full (Gym)
   - 4 → Upper/Lower ×2 or PPL + weak-point
   - 5 → PPL + Upper/Lower, or PPL + Skill + Weak-point (Calisthenic)
   - 6 → PPL ×2
   
   Each split returns an ordered sequence of `DayTemplate`s (e.g., `.push, .pull, .legs, .rest, .push, .pull, .legs`).

3. **Schedule onto the week.** Map `DayTemplate`s onto `trainingDays`. Rest days fill gaps. Two weeks = pattern × 2 with progression state carried forward.

4. **Fill each workout from the catalog.** Per `DayTemplate`:
   - 1–2 primary compounds (style/equipment-filtered)
   - 2–3 secondary exercises targeting day's muscle groups
   - 2–3 accessories — **weak-point biasing lands here**
   - **B-bias (volume):** if `focusAreas` priority-1 is shoulders, add an extra shoulder accessory on push day + extra rear-delt accessory on pull day.
   - **C-bias (selection):** when two equivalent exercises are candidates, pick the one whose `biasedMuscleGroups` includes the weak point (incline DB press over flat bench for front-delt bias).
   - Respect `ExercisePreferences`: skip `.no`, prefer `.yes`, swap `.sub` to the user's listed alternative.
   - **Exercise refresh rule:** for each slot in the workout, if the same exercise has been prescribed for ≥ 3 consecutive blocks *and* no tier unlock has fired for it *and* it hasn't gone through a plateau-deload, rotate to a same-pattern alternative from the catalog (e.g., barbell row → Pendlay row → seal row). Record the rotation in `ProgramBlock.exerciseRotationsThisBlock`. Progression state carries to the new exercise via family mapping when available; otherwise seeds conservatively.

5. **Set prescriptions from progression state.**
   - Existing `ProgressionState` → use `currentWorkingWeightKg` + `targetRepMax`.
   - New exercise → seed from equivalent-tier state or start conservative from experience level.
   - `targetRPE` comes from `trainingFeedbackMode`: `silent = 0` (always passes), `quick = 7` (Accumulation target), `detailed = 7` with per-set values captured when the user taps each set's chip.

6. **Generate macros.** Pure math:
   - Maintenance TDEE × (0.85 if cut else 1.0)
   - Protein: 1.8 g/kg (2.2 g/kg if cut)
   - Fat: 25% of calories
   - Carbs: remainder

7. **Generate recovery.** Per day, 2–3 mobility/stretch items from a recovery catalog keyed to the day's muscle groups. Rest days get a full mobility flow.

8. **Generate rationale.** Populate `ProgramRationale.decisions` with actual inputs: "4 days/week → Upper/Lower × 2", "Scan flagged narrow shoulders → added lateral raise + face pull emphasis", "Bodyweight only → swapped barbell rows for inverted rows", etc. Surfaced by existing `WhyThisProgramView`.

9. **Write `ProgramBlock`** with `accessoryBias`, `cutModeActive`, `scanId`. Detect continuity via `previousBlock`.

**Code locations affected:** `LocalProgramGenerator.swift`, `ProgramBuilder.swift`, `ProgramPhaseEngine.swift` refactored into this pipeline. `ProgramGenerationService.swift` (LLM path) deprecated — keep file, stop calling.

## 5. Progression + Block Rollover Loop

### During the block (days 1–14)

1. User logs `WorkoutLog` → `WorkoutLogService.saveLog` fires `ProgressionEngine.ingest(log:)` (existing, unchanged).
2. Engine per-exercise:
   - Top-of-range + target RPE for 2 consecutive sessions → weight bumps → `.progressionAdvanced` event (home toast).
   - Top-of-range + target RPE meets a tier gate → tier unlocks → `.tierUnlocked` event (feat card).
   - `PlateauDetector` sees no progress for N sessions → flags for deload on next rollover.
3. `TrainingFeedbackMode = silent` → engine treats `targetRPE = 0`, RPE check always passes. Pure rep-based double progression runs transparently.

### At block boundary (day 14 → day 15)

1. App detects block complete (last day logged or 14 days elapsed).
2. **Re-scan nudge** on home screen: "Block complete — want to check in?" Dismissible, not modal. Deep-links to scan flow. Scans and progress photos are both available; the scan is the primary CTA at this beat.
3. **Next block generates regardless of re-scan.**
   - **Accessory-bias refresh is gated:**
     - If user re-scanned AND the top-2 `focusAreas` (by priority) differ from the previous block's — refresh `accessoryBias` from the new scan. Set `biasRefreshedFromPrevious = false`.
     - Otherwise (no scan, OR scan priorities unchanged) — carry forward previous block's `accessoryBias`. Set `biasRefreshedFromPrevious = true`. *This is the common case and the intended behavior.*
   - Split structure: unchanged unless user edited `targetFrequency` or `trainingDays` in Settings.
   - Exercise prescriptions pulled from current `ProgressionState` store.
   - Tier-unlocked exercises replace predecessors (e.g., unlocked weighted pull-up → next block prescribes weighted pull-up).
   - **Exercise refresh** (see Section 4, step 4) rotates stale exercises per the ≥3-block rule.
   - Plateaued exercises (≥1 flagged by `PlateauDetector`) run in **per-exercise deload** for this block: weight × 0.85, `targetRPE = 6`, reps/sets unchanged. One block only, then auto-return to Accumulation.
4. New `ProgramBlock` record written, `blockNumber += 1`, linked to previous, with `biasRefreshedFromPrevious` and `exerciseRotationsThisBlock` populated.
5. Home updates to "Day 1: [Label]." (Block number hidden per Section 6 UI spec — only appears in the Progress Album as "Arc N.")

### Cut mode inside the loop

- `CutMode.enabled == true`:
  - Macros: 15% deficit, protein 2.2 g/kg.
  - `ProgressionEngine` switches to **preserve mode**. Weight-bump events short-circuit (held instead of advanced). Tier unlocks still fire (user can still hit reps/skill targets — those unlock regardless of weight).
  - Volume unchanged. No punishment.
- **8-week soft cap:** `CutMode.startedAt` > 8 weeks → home card: "Your cut is 8 weeks in. Want to take a maintenance break?" Dismissible. Not forced.
- User toggles Cut off → progression resumes automatically on next block.

## 6. UI Surfaces

### Home screen (`ProgramOverviewView`)

- **Hero = today's workout card.** Large, top of screen.
  - Rest day: "Rest Day — Recovery Flow" with mobility items.
  - Training day: `"Day 4 — Push + Shoulder Bias"`, exercise chips, `Start Workout` CTA.
  - **No block number shown here.** Day-to-day is jargon-free.
- **2-week calendar strip** under hero. Horizontal scroll, today highlighted, completed days checkmarked. Tap → `DayDetailView`.
- **Status card** under calendar: days remaining in block, Cut mode badge if active, re-scan nudge badge at block end.
- **Progress toasts** from the top — tier unlocks, weight bumps, re-scan nudges.

### `DayDetailView`

- Keep current structure.
- Add a small **"Why this day"** line at the top when weak-point bias applies: e.g., "Extra shoulder work — your scan flagged narrow shoulders for V-taper." Sourced from `ProgramRationale.decisions`.
- Exercise cards show prescribed weight × reps, **Deload label** if active for that exercise, link to catalog entry (form demo, substitution sheet).

### Macros view (nutrition tab)

- Daily calorie, P/C/F targets. Cut badge if active.
- Targets only — not a tracker.
- Cut copy: "Cut mode — we've paused lift progression while you lean out. Resume anytime."

### Progress Album (new surface; lives in Profile tab)

- Grid of `ProgressPhoto`s.
- **Scan entries visually distinct** (small sparkle icon + arc counter "Arc 3 • Week 2").
- Comparison mode: tap two → side-by-side.
- Top CTAs: "Take progress photo" (primary) + "Full scan" (secondary; becomes primary when block is complete).

### Settings additions

- Training Feedback (Silent / Quick / Detailed) — default from experience, editable.
- Training Style — shows archetype default, override available.
- Training Days — Mon–Sun multi-select, count-locked to `TargetFrequency`.
- Cut Mode — toggle + 8-week soft-cap banner.

### Not included in MVP

- No separate meal-plan screen or recipes (scope).
- No RPE tutorial modal — UX uses plain-language `Chill / Locked / Grinding`.
- No phase/cycle visualization. User never sees Hawks jargon.

## 7. Onboarding Changes

- **Remove:** current-`Frequency` step (redundant).
- **Expand:** `Step14_Equipment` chips from 4 to ~7 (`Bodyweight only / Pull-up bar / Dumbbells / Barbell + rack / Bench / Cables or machines / Full gym`).
- **Keep:** `TargetFrequency` step.
- **Add:** `Step_TrainingDays` — Mon–Sun multi-select, validation requires count == `TargetFrequency`.
- **Keep:** Experience step (silently drives default `TrainingFeedbackMode`).
- **No new RPE question.** Default from experience, editable in Settings.
- **No new Cut question.** Defaults off, user toggles later.
- **No new Style question.** Archetype default, override in Settings.

Net: –1 step, +1 step. Onboarding length unchanged.

---

## 8. Migration / Rollout

Existing users with a generated `TrainingProgram` are **not blown away**.

- On next block boundary, the new generator runs with their existing inputs.
- Their `ProgressionState` carries forward untouched.
- Users who onboarded before this change lack `TrainingDays` and `TrainingStyleOverride`. On first app open post-update, a one-screen prompt fills in the gap: "Quick setup — which days do you train?" User picks, app proceeds.

## 10. Success Criteria

The redesign is complete and successful when:

1. A new user finishes onboarding and sees a populated 2-week program whose exercises demonstrably reflect their scan's weak-points, their equipment, and their selected training days.
2. Logging workouts advances weights per Hawks' rules; tier unlocks fire; home-screen toasts surface progression events.
3. At day 14, the app prompts a re-scan (dismissible); regardless of re-scan, a new block generates on day 15 with progression state carried forward.
4. **Accessory bias only refreshes on meaningful scan priority change.** When a user re-scans and their top-2 weak-point muscles are unchanged, the new block's accessory bias is identical to the previous block's. `ProgramBlock.biasRefreshedFromPrevious` is `true` in the common case.
5. **Exercise refresh rotates stale slots.** An exercise that's been prescribed for 3 consecutive blocks with no tier unlock and no plateau deload is rotated to a same-pattern alternative in the next block. `ProgramBlock.exerciseRotationsThisBlock` records these rotations.
6. Toggling Cut mode on shifts macros to a 15% deficit and pauses weight progression (tier unlocks still allowed), with no volume reduction. An 8-week soft-cap banner appears when applicable.
7. Settings offers Training Feedback, Training Style, Training Days, and Cut Mode — all editable post-onboarding.
8. Progress Album shows both daily progress photos and scan entries, visually distinct, comparable side-by-side.
9. No UI surface shows "phase," "cycle," "accumulation," "intensification," or RPE numeric jargon. Block number appears only in the album as "Arc N."
