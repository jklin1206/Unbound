# Local-Authoritative Program Store — Design Spec

**Date:** 2026-05-17
**Branch:** `program-redesign` · UNBOUND iOS
**Type:** Architecture — program data layer (perf + durability)
**Relation:** Unblocks the "instant Home/Program" goal; substrate for sub-projects C/D (skill→program adaptation).

## Context

The `TrainingProgram` is a ~monthly artifact (generated by Claude, replaced
on block rollover/regenerate) but is treated as volatile live data: every
app open does `fetchProfile` → `database.read` the **entire 84-day blob** →
`fetchLogs` (3 serial network hops; `ProgramViewModel.loadProgram`), with **no
launch-surviving on-device copy**. Consequences:

1. **Slow:** 2–3 cold network round-trips + full-blob `Codable` decode to show
   *today's* one workout, every launch.
2. **Data-loss bug:** the program is user-editable in memory
   (`ProgramViewModel.swapExercise`, `updateSetsReps`) and flushed as a
   whole-row upsert by `saveProgram()` (`database.create(program,
   documentId:)`). If that network write fails (offline/flaky), the edit is
   **silently lost on the next cold launch** — nothing persisted it locally
   and the next read pulls the stale remote row.

The fix is to make the device hold a durable, authoritative copy: the program
becomes a **local-first document** that is the fast-read source *and* the edit
surface; remote is sync/backup + the source of monthly replacements.

`SupabaseProgramService` already implements the remote side
(`saveProgram(_:userId:)` upsert + `current_program_id` patch;
`fetchProgram(id:)` remote-with-local-fallback) but **the UI does not call it**
— reads still go through `services.database` directly. This spec wires a real
local store and routes program reads/writes through it.

## Locked decisions (from brainstorming dialogue)

1. **Local-authoritative.** An on-device persisted copy of the current
   `TrainingProgram` + its `programId` is the source of truth the UI renders
   and edits. Reads are local-first and synchronous-fast; the network is
   revalidation/backup, never on the critical path.
2. **Write-through, durable-first.** Every program mutation (manual edit, swap,
   sets/reps, coach actions, future skill→program) writes the full program to
   the local store **immediately** (so an edit is never lost), then enqueues a
   best-effort background remote sync with retry via
   `SupabaseProgramService.saveProgram`.
3. **Invalidation = replace-on-new-`programId`.** The store records the local
   program's id. A *new* program (block rollover / regenerate) produces a new
   `programId` and bumps `users.current_program_id`; that — and only that —
   supersedes the local copy. A remote write to the **same** id never clobbers
   local (local edits win). Detection is a cheap id comparison
   (`current_program_id` vs the locally stored id), not a blob fetch; the full
   program is fetched only when the id actually differs.
4. **Coach is on-device, no special-casing.** `CoachActionExecutor` runs
   in-process and routes program changes through the same in-memory
   `ProgramViewModel` + this store (its permanent changes return as a new
   `programId` via the `ExercisePreference`→next-generation path). No separate
   coach reader/writer exists, so no cross-writer conflict on the same id.
5. **Cross-device is explicitly deferred.** Two devices editing the same
   `programId` will not see each other's same-id edits until a new program.
   Accepted for a single-active-device app now. The clean upgrade path
   (non-breaking) is a `version`/`updatedAt` token on the row to detect same-id
   remote changes; the store is designed so adding it later is additive.

## Architecture

### `ProgramStore` (new — `UNBOUND/Services/Program/ProgramStore.swift`)

`@MainActor final class ProgramStore: ObservableObject` — the single owner of
the active program on device. Pattern mirrors the proven `WorkoutDraftStore`
(Codable snapshot to an app-support file).

State / API:
- `@Published private(set) var program: TrainingProgram?`
- Persisted file: `Application Support/program-store.json` holding
  `Cached { program: TrainingProgram; programId: String; userId: String;
  syncedAt: Date?; dirty: Bool }`.
- `func loadLocal(userId:) -> TrainingProgram?` — synchronous decode of the
  cached file if `userId` matches; sets `program`; returns it (nil if none).
- `func revalidate(userId:expectedProgramId:) async` — given the authoritative
  `current_program_id` (the caller already learns this from the profile/the
  cached value): if it differs from the cached id → `SupabaseProgramService
  .fetchProgram(id:)` the new program, replace + persist local, clear `dirty`.
  If same id → no-op (local wins). Never overwrites a `dirty` local program
  with a same-id remote.
- `func save(_ program: TrainingProgram, userId:) async` — write-through:
  persist locally + set `dirty=true` synchronously **first**, publish, then
  `await SupabaseProgramService.saveProgram(program, userId:)`; on success
  clear `dirty`, set `syncedAt`. On failure stay `dirty` (retried on next
  `save`/`flushIfDirty`).
- `func flushIfDirty(userId:) async` — opportunistic retry of an unsynced
  local program (called on app foreground / next load).
- `func clear()` — wipe on sign-out (parallels `WorkoutDraftStore.clear`).

`ProgramStore` is added to `ServiceContainer` as `services.programStore`
(constructed once, like other services).

### Integration points (route through the store; minimal diffs)

- **`ProgramViewModel.loadProgram(programId:)`** — first `services.programStore
  .loadLocal(userId:)`; if a program is present render it immediately
  (`state = .loaded`), then `await services.programStore.revalidate(
  userId:expectedProgramId: programId)` and update if it replaced. Only when
  no local copy exists does it block on a network fetch (first run).
- **`ProgramViewModel.saveProgram()`** — replace the direct
  `services.database.create(...)` with `await services.programStore.save(
  program, userId:)` (durable-first + background remote). Same external
  behavior, now crash/offline-safe.
- **`UnboundHomeView.loadProfileAndProgram` / `ProgramOverviewView.task`** —
  use `programStore.loadLocal` for the instant paint; the existing parallel
  profile fetch supplies `current_program_id` for `revalidate`. Generation
  still only on genuine first run (`current_program_id == nil`), unchanged.
- **`ProgramGenerationService`** (first-run/rollover) — after producing a
  program, seed the store (`save`) so the just-generated program is the local
  authority immediately (it already persists remote; this adds the local
  authority + removes the re-read).

### Reused (do not rebuild)

- `SupabaseProgramService.saveProgram` / `fetchProgram(id:)` — the entire
  remote + not-authenticated/local fallback path. `ProgramStore` calls these;
  it does not talk to Supabase directly.
- `WorkoutDraftStore` file-persistence pattern (Codable → app-support file,
  `clear()` on sign-out) — copy the approach, not the file.
- `TrainingProgram` Codable shape — unchanged (no schema/migration).

### Unchanged

`TrainingProgram`/`ProgramDay` models, generation logic, block rollover
(`BlockRolloverService` still produces a new id → that's the replace trigger),
coach action semantics, the program-persistence detached-write fix
(`ProgramGenerationService` `41ff444`), the load-perf parallelization
(`25af5ab`/`8478f21`), tokens/UI.

## Testing / verification

- Unit (`UNBOUNDTests/Services/ProgramStoreTests.swift`, mock
  `SupabaseProgramService`/file via temp dir):
  - `save` then `loadLocal` round-trips the program; survives a fresh store
    instance (durability).
  - `save` with a failing remote keeps `dirty=true` and the program still
    loads locally (the data-loss-bug regression test).
  - `revalidate` with the **same** `programId` is a no-op even if remote
    differs (local-wins) and never clears a `dirty` program.
  - `revalidate` with a **new** `programId` fetches + replaces + persists +
    clears `dirty`.
  - `loadLocal` returns nil for a different `userId` (no cross-account leak);
    `clear()` wipes.
- `xcodebuild … build` → `** BUILD SUCCEEDED **`; `xcodebuild … test` → all
  green except the known pre-existing `FriendChallengeServiceTests`/
  `SquadMissionServiceTests` RLS flap; zero NEW failures. SourceKit
  cross-file noise ignored.
- Parity: a logged/edited session still produces the same remote `programs`
  row eventually; `saveProgram`'s observable behavior is unchanged on success.
- On-device (jlin): cold-open Program → today's workout paints instantly from
  local (no spinner); edit an exercise offline → kill app → reopen → **edit is
  still there** (was the bug); force a rollover/regenerate → new program
  replaces local.

## Out of scope

Cross-device same-id sync (deferred — future `version` token), per-field
program merge, partial/lazy day loading, server-side composed endpoints,
the broader Home read-through cache for non-program domains, sub-projects
C/D themselves (this only provides the durable mutable substrate they need).

## Execution

Subagent-driven (fresh subagent per task, spec-then-quality review at the
cluster seam), Sonnet minimum, frequent commits (scoped `git add`, no
`git add -A`), co-authored trailer, on-device sign-off by jlin.
