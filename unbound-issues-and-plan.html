02 — Definition of done

No fix is "done" on a claim — only on a passing proof

THE RULE

Before touching code we write the falsifiable done-when first. Engine-math changes re-run the persona sim so balance shifts are visible, not assumed. The proof type is matched to the issue:

LOGIC
  • Unit / concurrency test — assertion on the rule
  • Integration test — a full multi-service flow
  • CI check — green required gate on PR
BEHAVIOR
  • Sim re-run + screenshot — cohort matrix shifts as predicted
  • Analytics funnel — per-step drop-off in PostHog
VISIBLE
  • UI screenshot in the simulator — the state actually renders
  • Instruments trace — frame-time / no main-thread stall

06 — How we'll actually do the work

Guardrails for every fix

PROCESS
  • Write the done-when test first — it must fail before the fix and pass after (esp. migration, Coach, date bugs)
  • One issue → one atomic commit, traceable to its register ID
  • Engine change → re-run the persona sim + attach the updated cohort screenshot so balance shifts are seen, not assumed
GUARDRAILS
  • Stay on the unified spine — all training through PerformanceLog → TrainingCompletionService
  • No new parallel reward/logging paths — consolidate, don't fork
  • Migrations reversible & idempotent; never destructive without a proven backup path

Each phase closes with: green CI · re-run sim · empty failing-proof list · a short handoff note. The next phase starts from verified ground.
