01 — The premise

A real strength program, wearing a JRPG progression sheet.

THE LOOP

Train → earn proof → see your character grow → pass a trial → rank up → repeat.

Every completed set feeds one pipeline that ripples into XP, attributes, skill ranks, body-map heat, badges, and trial readiness. The fantasy is an honest fitness app that feels like leveling a shonen protagonist.

THE SPINE

All training converges on one route:

PerformanceLog→ TrainingCompletionService→ side effects

XP · AP · attributes · skill tiers · body map · badges · trials — all fan out from one completion event.

05 — Program generation

A deterministic engine, not an AI coach

Onboarding + Calibration→ DeterministicProgramGenerator→ 28-day Arc→ 2×14-day Waves→ Days → Workouts
THE ARC = 28 DAYS
  • Days 1–14 Accumulation · 15–21 Intensification · 22–24 Realization · 25–28 Deload
  • Wave 2 (day 15+) bumps RPE +1 on engine-owned days
  • Block ends → rollover regenerates a fresh Arc, carrying working weights forward & rotating stale exercises
DAILY RESOLVER

Each day the stored workout is reshaped by modifiers in order:

substitution → trial-prep → skill-taper → deload → short-session

It reshapes the prescription — it does not recompute weights. Weights live in per-exercise ProgressionState.

06 — How the weight actually moves

Double progression, RPE-gated

THE RULE

Hit top-of-rep-range at or below target RPE for 2 sessions in a row → advance.

  • Compounds: bump weight (+5kg lower / +2.5kg upper), reset the counter
  • Accessories: add 2 reps to a ceiling, then bump weight
  • Bodyweight skills: no load bump — defers to the skill tree
SAFETY VALVES
  • Plateau: exercise stalls ≥3× in 30 logs with no advance
  • Deload: triggers at ≥2 plateaus or week-4 of a block → 2 easy sets @ RPE 6
  • Coach card: one-tap swap / deload / rep-range / acknowledge — each reversible

Note: auto-deload is surfaced as a suggestion, not silently forced — see gaps.

Per-set AP fans out into three dopamine cadences

PER SET

Logarithmic on reps → cheap to start, brutal to scale. A 10-rep set ≈ 28 AP; doubling to 20 reps only ≈ 35 AP.

08 — Identity systems

Skill tree & attribute hex

SKILL TREE

~125 live nodes across 9 clusters (pull, push, core/levers, legs, handstand, planche, one-arm…).

  • Nodes unlock by proving prereqs — OR-across-groups, AND-within
  • Each node carries its own 9-tier rank, computed from log history
  • Clusters can gate behind another cluster's keystone
ATTRIBUTE HEX

6 axes that draw your build shape.

  • Every exercise carries a weight vector → AP fans into axes
  • Novelty bonus (×1.0–1.5): training cold body regions pays more
  • Shape resolves to balanced / hybrid / specialist / lean

The hex is the "what kind of athlete am I" read; lift & skill ranks answer "how good at X"; Overall LV answers "how long have I trained."

10 — Proof you're changing

Body scan & monthly checkpoints

THE SCAN

Photos are visual proof only — the body is never graded.

  • Captures one front photo, stored on-device
  • Build Identity is read from the attribute system, not the image
  • Onboarding uses on-device Apple Vision pose detection — no upload, no server AI verdict
CHECKPOINT (per Arc)

End-of-Arc questionnaire → validated signals (recovery, missed sessions, cardio, pain, nutrition).

Produces a deterministic loadAdjustmentBias and a Claude-written recap paragraph.

Scans only ever surface positive deltas — "scans never show setbacks." Decay caps idle axes quietly.

15 — The shape of the year

Fast for three months, then a hard taper

The log-based AP formula + concave level curve + no decay = a deliberate "honeymoon then grind."

Novice → Forged in ~3 months feels electric. Then the attribute display score (which gates the upper trials) climbs at 1/50 the rate of raw XP — so progress visibly decelerates exactly when the user expects to keep flying.

SCAN SIDE (MONTHLY)
  • Month 1: "your arc begins," no delta
  • Month 2: first positive delta → "build identity resolved" badge
  • Months 3–12: shape migrates balanced → specialist as a dominant axis pulls ahead
THE CATCH

The checkpoint questionnaire feels like it reshapes next month's load — but the bias it computes is consumed only as rationale copy. The next Arc clones the prior Arc's sets verbatim. Adaptation is largely cosmetic.

What's Missing & What Needs Work

The app is genuinely production-wired. The gaps are mostly loops that compute a signal but never close it — plus a few honest stubs.

UNBOUND // PART III
16 — High impact

Three open loops that change the product story

1 · CHECKPOINT BIAS NOT APPLIED

Recovery/missed/cardio/pain all compute a clamped loadAdjustmentBias — then ArcGenerator only uses it to pick a sentence. The next Arc copies the old sets/reps/RPE unchanged. "Your answers reshape your program" is narrative-only. ArcGenerator.expandToArcDays

2 · RANK DECAY IS A NO-OP

By design "nothing decays," but RankDecayService.applyDecay does nothing — it only flips a cosmetic "recalibrating" banner and resets a streak counter. The honest staleness model (recent-vs-lifetime, stale flags) is specced but not wired. RankDecayService.swift:108

3 · NO LLM COACH EXISTS

PTContextBuilder assembles a full athlete dossier for a coaching chat — but nothing calls it. If a conversational PT is the intended product, it's unbuilt scaffolding. PTContextBuilder · AISessionGeneratorService (retired stub)

17 — Medium impact

Loops that under-deliver

  • Skill auto-proof is half-built. Reps & weight auto-detect; holds, levers, carries, planche always return false → the L-sit/front-lever/handstand half of the tree needs a manual "I hit this" tap.
  • Auto-deload isn't automatic. The detector exists but only fires from a Coach card tap — a plateauing user gets no intervention unless they go looking.
  • Realization/Peaking unreachable. Gated by rank in the spec, but the live generator assigns phase by day-number and never produces peaking at all.
  • Badge catalog ↔ service mismatch. Service awards ids (sessions_25, streak_14) absent from the catalog; catalog badges (streak_30/100) are never awarded. Some show blank, some unreachable.
  • Squad polish deferred. Roster shows raw UUIDs instead of handles; presence is 30s polling; leave-squad captain handoff has a known race; missions need a server cron.
  • Scan delta carries dead columns. Before/after body-part grades are frozen at a neutral 5→5 — persisted but inert.

19 — Where I got confused (and a new user will too)

The naming maze

3 "XP"s, 3 "RANK"s
  • Session XP (a counter) ≠ Overall LV XP ≠ Skill-node XP — three curves, one word
  • SubRank (E–S strength) ≠ SkillTier (per-node) ≠ Overall Rank (the headline)
  • RankState.swift is empty — a trap when reasoning about "rank"
TWO "BLOCK"s
  • The generator's positional Arc-day phase vs. per-exercise ProgressionState.blockType vs. a log-derived PhaseEngine — three unsynced notions
  • So the screen can show "Realization 3–5" while the bump engine still keys off accumulation ranges
  • "Trials" means two different systems: rank gates and weekly Vows

These aren't bugs so much as vocabulary debt from a fast migration. They're the #1 thing I'd document before onboarding a new engineer — or a new user.

20 — Subtle landmines

Things that look right but aren't

  • Shape uses peak, delta uses current. Build Identity reads attribute peaks; the scan delta reads current. With decay, a scan can say "held the line" while your displayed identity hasn't moved — intentional-looking, genuinely subtle.
  • Rank-up notifications key off a per-launch in-memory baseline that resets to E− every app start — so cinematics and rank badges can re-fire or mis-fire across sessions.
  • Pin ordering is load-bearing. Scan-delta correctness depends on a snapshot being taken before reading history[n-2]. Reorder it and deltas silently corrupt.
  • DEBUG builds unlock all features — easy to mistake "paywall isn't working" for a bug when it's a dev flag.
  • Squads are Release-strict: an anonymous (not-signed-in) user can't create or join one, with no in-UI explanation.

PART IV

How I'd Write the Tutorial

The systems are deep — but a first-time user should never feel the depth. Teach the loop, hide the machinery, reveal complexity only as it's earned.

25
UNBOUND // PART IV
21 — The principle

Don't explain the RPG. Let them feel the first level-up.

DON'T
  • Front-load AP vs XP vs LV vs rank
  • Show the 9-tier ladder before they've earned rung 1
  • Explain attribute vectors or novelty math
DO
  • Get them to one completed set fast, then fire the reward sequence
  • Name the loop in 6 words: train, earn, grow, prove, rank up
  • Surface each system the first time it actually triggers
FRAME

Anime training-arc voice per the brand guardrail: "Start your arc", never "fix what's wrong with you." Invite, don't diagnose.

23 — Format & what I'd fix first

Interactive, in-product, earned

FORMAT
  • In-product coach-marks, not a separate manual — teach at the moment of first trigger
  • A persistent "What is this?" affordance on every rank/XP/attribute chip for the curious
  • A one-page glossary card that disambiguates AP / LV / Rank / Tier in plain words
  • A short "your week" recap that names the loop out loud after session 1
PREREQS FOR A GOOD TUTORIAL

Before teaching, fix what would confuse a learner:

  • Unify the XP/rank vocabulary (or alias it consistently in UI)
  • Make the checkpoint actually change load — or stop implying it does
  • Give hold/lever skills a clean proof path so the tree feels alive
