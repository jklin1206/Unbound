Finding 01

The workout system is now unified

  • TrainingSessionDraft is the planned session source.
  • TrainingBlock carries strength, bodyweight, skill, cardio, carry, routine, custom.
  • ActiveWorkoutSession is the live logging model.
  • PerformanceLog is the richer completed-session record.
Finding 02

The reward code is still split-brained

  • The premium WorkoutRewardSequenceView exists.
  • But WorkoutLoggingViewModel.makeRewardSequenceSummary() still fabricates some reward data.
  • The newer ActiveWorkoutContainerView still presents the older RewardCelebrationView.

Conclusion: the visuals are ahead of the data integration.

Critical Point

Rewards should attach to PerformanceLog

PerformanceLog preserves the exact context rewards need:

  • Block type: skill, strength, bodyweight, custom, carry, routine.
  • skillId for skill-tree work.
  • Actual reps, weight, hold seconds, RPE, skipped state.
  • Session source: program, skill, custom, routine.
Integration Plan

Add one resolver service

WorkoutRewardSequenceResolver

snapshotBeforeCompletion(performanceLog, userId)
resolveAfterCompletion(snapshot, performanceLog, userId)
  → WorkoutRewardSequenceSummary

This keeps reward logic out of the SwiftUI view and out of the old logging view model.

Flow

Exact completion sequence

  1. Assemble PerformanceLog.
  2. Capture reward snapshot before save.
  3. Call TrainingCompletionService.complete(...).
  4. Let existing services update progression, skills, attributes, badges.
  5. Resolve after-save diff into WorkoutRewardSequenceSummary.
  6. Present WorkoutRewardSequenceView.
Exercise Classification

Reward each type honestly

Ranked lift

Use LiftTierCriteria.table. Copy: Rank Progress / Promotion.

Custom exercise

No fake rank. Use PRs and history. Copy: New Best / Rep PR / Load PR.

Attributes

Diff real AttributeService state before vs after.

Standards

No hardcoded reward thresholds

  • Skill standards come from SkillNode.tierCriteria.
  • Lift standards come from LiftTierCriteria.table.
  • The reward UI receives only resolved progress and tier changes.

If standards change later, current progress recomputes from the new source of truth.

Model Cleanup

Broaden lift rewards

LiftProgressReward is too narrow now. Use a general progression reward:

ProgressionRewardSource:
  skill(skillId)
  liftStandard(exerciseName)
  customExercise(exerciseName)

The page can say Progression Reveal, while each reward uses accurate language.

Bottom Line

Yes — it is easier now

The unified workout model already carries the context. The missing piece is a resolver that diffs real progression state before and after completion, then feeds the existing cinematic reward UI.

Do not make the reward screen decide standards. Make it display resolved rewards.
